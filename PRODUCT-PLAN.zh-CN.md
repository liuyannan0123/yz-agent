# AI 知识助手产品方案

## 目标

把当前项目从“本地可用的 RAG 系统”升级为“用户可上传文件、稳定提问、可持续推广”的产品。

产品目标：

- 用户上传文件后可直接提问
- 常见知识问答尽量控制在 `3 秒内`
- 回答优先基于命中内容，不胡编
- 用户不会代码也能长期使用
- 新文件类型进来后，系统先能用，再逐步增强

## 当前已具备的能力

当前系统已经实现了以下能力：

- 中文 Web 问答页面
- 中文助手“小六”人设
- 发送后的呼吸等待态
- 文件上传、提取预览、确认入库
- OCR 图片预览后入库
- 原文件保存到知识目录
- 向量写入 `rag_vectors`
- `SHA256` 内容去重
- 混合检索
- 部分快答能力

当前已经上线的快答类型：

- 流程/标题类快答
  - 例如：`公文处理流程`
- 人物属性类快答
  - 例如：`电话`、`邮箱`、`学校`、`地址`
- 时间/记录类快答
  - 例如：`最近一次工作是什么时候`
- 已覆盖的推荐类快答
  - 例如：`适合做什么`、`适合什么岗位`

## 当前核心问题

即使已经能用，当前仍有 4 类需要持续收敛的问题：

1. 检索召回稳定性
- 中文短语拆分不足时，文档明明入库了，也可能答不到
- 标题、文件名、正文标题命中没有被充分利用时，容易退回普通聊天

2. 多文档歧义
- 相似文件变多后，容易被错误来源干扰
- 同一个问题在多人简历、多份相似资料场景下，可能缺少明确对象定位

3. 事实问答与总结问答混用
- 电话、学校、日期、流程步骤这类问题更适合结构化直返
- 如果继续交给模型总结，就会变慢，也更容易出错

4. 扩展能力还未配置化
- 当前很多快答规则仍在代码里
- 后续应逐步改成页面配置，降低维护门槛

## 产品架构

建议将系统稳定在 4 层架构上：

1. 上传解析层
2. 混合检索层
3. 结构化抽取层
4. 回答路由层

### 1. 上传解析层

负责把用户文件变成可检索、可追溯、可更新的知识。

包含能力：

- 文档上传
- 图片 OCR 上传
- 提取预览
- 预览可编辑校对
- 确认后入库
- 原文件保留
- 去重
- 同文件覆盖更新

建议流程：

1. 用户上传文件
2. 系统提取文本
3. 返回提取预览
4. 用户可编辑校对文本
5. 确认后保存原文件和最终文本
6. 同步写入向量库与结构化事实表

### 2. 混合检索层

不要只依赖向量检索，统一采用 3 路召回：

- 标题/文件名召回
- 关键词召回
- 向量召回

然后统一重排。

通用能力应包括：

- 中文 query 拆分
- 保留整词
- 自动补子词
- 标题命中加权
- 文件名命中加权
- 文件级聚合重排
- 低置信度拒答

这一层主要解决：

- 文档已入库但检索不到
- 问法稍有变化就掉回普通聊天
- 新文件类型一来就失效

### 3. 结构化抽取层

不是所有文件都必须完整图谱化，但高频问答相关字段应优先抽取。

第一阶段重点支持：

- 简历类
- 工作日志类
- 流程/制度类

建议抽取字段：

简历类：

- 姓名
- 学校
- 专业
- 居住地
- 电话
- 邮箱
- 公司
- 岗位

工作日志类：

- 报告人
- 报告日期
- 工作内容
- 问题
- 解决方案

流程/制度类：

- 文档标题
- 章节标题
- 步骤序号
- 步骤内容

建议表：

- `rag_vectors`
- `rag_file_index`
- `person_profiles`
- `worklog_facts`
- `workflow_steps`
- `document_sections`

### 4. 回答路由层

问题不要再走单一路径，而是先判断问题类型，再选择回答链路。

建议问题类型：

- 标题/流程类
- 人物属性类
- 时间/记录类
- 推荐/岗位建议类
- 复杂事实类
- 总结/开放问答类

对应处理方式：

标题/流程类：

- 优先命中标题、文件名、章节标题
- 命中后直接提取步骤或段落

人物属性类：

- 优先查结构化字段
- 查不到再回退到 RAG

时间/记录类：

- 优先查日期和工作记录
- 支持“最新”“最早”“某一天”

推荐/岗位建议类：

- 优先基于简历里的已知能力和经历做短答
- 不进入长模型生成链路

复杂事实类：

- 先查结构化事实
- 再补 RAG
- 必要时由模型整合

总结/开放问答类：

- 走 RAG + 模型生成

## 当前快答机制说明

快答的核心不是“不要知识库”，而是：

- 先命中知识库
- 再直接从命中文本里抽字段或拼步骤
- 不走慢速 `ollama.chat` 长生成链路

所以：

- 快答 = 检索/抽取/模板返回
- 慢答 = 检索 + 模型生成

这也是为什么同样是“知识库回答”，有的问题是 `0.07s`，有的问题是 `10s+`。

## 当前快答规则范围

当前快答已覆盖：

1. 流程/标题类
- 触发词：`流程`、`步骤`、`规范`、`制度`、`办法`、`指引`、`手册`、`要求`、`规定`

2. 人物属性类
- 电话
- 邮箱
- 学校
- 专业
- 地址
- 公司
- 岗位

3. 时间/工作记录类
- `最新`
- `最近`
- `最早`
- `最后一次`
- `某月某日`
- `做了什么`

4. 推荐类
- `适合做什么`
- `做什么合适`
- `适合什么岗位`
- `适合什么工作`
- `推荐什么岗位`

已覆盖的推荐岗位方向：

- 运营/行政/活动/商务支持
- Java 开发
- 后端开发
- Java 后端开发
- 数据库开发/数据治理
- 项目管理
- 微服务开发

## 为什么有的人快、有的人慢

本质不是“换了一个人就变慢”，而是：

- 如果问题命中了快答规则，就会直接短答
- 如果问题没有命中快答规则，就会退回模型生成

所以“快慢差异”来自：

- 是否命中知识库
- 是否命中快答规则
- 是否仍然需要模型生成

## 页面配置化方向

后续不应把所有规则都写死在代码里，而应逐步做成页面配置。

适合配置化的内容：

- 快答触发词
- 属性别名
- 流程类关键词
- 推荐岗位映射
- 标题命中权重
- 是否启用快答
- 低置信度拒答开关

不适合完全配置化的内容：

- 核心检索算法
- 向量检索实现
- OCR 引擎
- 数据库存储逻辑
- 新的回答模式框架

一句话：

- 通用底座在代码里
- 经常变化的规则放到页面配置里

## 性能目标

产品目标不是“所有问题都在 3 秒内”，而是“高频核心问题在 3 秒内”。

建议目标：

- 人物属性类：`0.05s - 1s`
- 时间/记录类：`0.1s - 3s`
- 流程/标题类：`0.05s - 3s`
- 推荐类：`0.05s - 3s`
- 总结/开放问答类：允许更慢

提速关键：

- 高频事实问题不进大模型
- 结构化查询优先
- 只有需要总结时才走模型

## 开发优先级

### 第一阶段：通用检索底座

- 中文 query 拆分
- 标题/文件名召回
- 混合检索
- 知识启用门槛重构
- 低置信度拒答

### 第二阶段：高频快答

- `workflow_steps`
- `person_profiles`
- `worklog_facts`
- 推荐类快答

### 第三阶段：配置化扩展

- 快答配置页面
- 字段别名配置
- 岗位映射配置
- 标题权重配置

## 最小可推广版本

如果要形成可推广产品，最小版本至少要满足：

- 文件上传后可稳定预览
- 预览内容可编辑
- 同文件覆盖入库，不污染知识库
- 典型问题能基于来源回答
- 回答有来源和耗时
- 命中不准时不胡编
- 高频问题尽量快答

## 一句话总结

最终产品形态应是：

`上传解析 -> 文本预览与校对 -> RAG + 结构化双写入库 -> 问题先路由 -> 混合检索 -> 快答优先 -> 模型只做总结与兜底`

这是一套既能解决当前问题，也能支撑后续推广的真实方案。
