# AI 知识助手产品方案

## 目标

把当前项目从“本地可用的 RAG 系统”升级为“用户可上传文件、稳定提问、可持续推广”的产品。

产品目标：

- 用户上传文件后可直接提问
- 常见知识问答尽量控制在 `3 秒内`
- 回答优先基于命中内容，不胡编
- 用户不会代码也能长期使用
- 新文件类型进来后，系统先能用，再逐步增强

## 当前已具备的能力

当前系统已经实现了以下能力：

- 中文 Web 问答页面
- 中文助手“小六”人设
- 呼吸等待态
- 文件上传、提取预览、确认入库
- OCR 图片预览后入库
- 原文件保存到知识目录
- 向量写入 `rag_vectors`
- `SHA256` 内容去重
- 混合检索
- 部分快答能力

当前已经上线的快答类型：

- 流程/标题类快答
- 人物属性类快答
- 时间/记录类快答
- 已覆盖的推荐类快答

## 当前核心问题

当前仍有 4 类问题需要持续收敛：

1. 检索召回稳定性
- 中文短语拆分不足时，文档明明入库了，也可能答不到
- 标题、文件名、正文标题命中没有被充分利用时，容易退回普通聊天

2. 多文档歧义
- 相似文件变多后，容易被错误来源干扰
- 同一个问题在多人简历、多份相似资料场景下，可能缺少明确对象定位

3. 事实问答与总结问答混用
- 电话、学校、日期、流程步骤这类问题更适合结构化直返
- 如果继续交给模型总结，就会变慢，也更容易出错

4. 扩展能力仍未配置化
- 当前很多快答规则仍在代码里
- 后续应逐步改成页面配置，降低维护门槛

## 产品架构

建议将系统稳定在 4 层架构上：

1. 上传解析层
2. 混合检索层
3. 结构化抽取层
4. 回答路由层

### 1. 上传解析层

负责把用户文件变成可检索、可追溯、可更新的知识。

包含能力：

- 文档上传
- 图片 OCR 上传
- 提取预览
- 预览可编辑校对
- 确认后入库
- 原文件保留
- 去重
- 同文件覆盖更新

### 2. 混合检索层

不要只依赖向量检索，统一采用 3 路召回：

- 标题/文件名召回
- 关键词召回
- 向量召回

然后统一重排。

通用能力应包括：

- 中文 query 拆分
- 保留整词
- 自动补子词
- 标题命中加权
- 文件名命中加权
- 文件级聚合重排
- 低置信度拒答

### 3. 结构化抽取层

高频问答相关字段应优先抽取。

第一阶段重点支持：

- 简历类
- 工作日志类
- 流程/制度类

建议抽取字段：

简历类：

- 姓名
- 学校
- 专业
- 居住地
- 电话
- 邮箱
- 公司
- 岗位

工作日志类：

- 报告人
- 报告日期
- 工作内容
- 问题
- 解决方案

流程/制度类：

- 文档标题
- 章节标题
- 步骤序号
- 步骤内容

### 4. 回答路由层

问题不要再走单一路径，而是先判断问题类型，再选择回答链路。

建议问题类型：

- 标题/流程类
- 人物属性类
- 时间/记录类
- 推荐/岗位建议类
- 复杂事实类
- 总结/开放问答类

对应处理方式：

- 标题/流程类：标题命中后直接提取步骤或段落
- 人物属性类：优先查结构化字段
- 时间/记录类：优先查日期和工作记录
- 推荐/岗位建议类：优先基于简历里的已知能力和经历做短答
- 复杂事实类：先查结构化事实，再补 RAG，必要时由模型整合
- 总结/开放问答类：走 RAG + 模型生成

## 当前快答机制说明

快答的核心不是“不要知识库”，而是：

- 先命中知识库
- 再直接从命中文本里抽字段或拼步骤
- 不走慢速 `ollama.chat` 长生成链路

所以：

- 快答 = 检索/抽取/模板返回
- 慢答 = 检索 + 模型生成

## 页面配置化方向

后续不应把所有规则都写死在代码里，而应逐步做成页面配置。

适合配置化的内容：

- 快答触发词
- 属性别名
- 流程类关键词
- 推荐岗位映射
- 标题命中权重
- 是否启用快答
- 低置信度拒答开关

不适合完全配置化的内容：

- 核心检索算法
- 向量检索实现
- OCR 引擎
- 数据库存储逻辑
- 新的回答模式框架

## 性能目标

产品目标不是“所有问题都在 3 秒内”，而是“高频核心问题在 3 秒内”。

建议目标：

- 人物属性类：`0.05s - 1s`
- 时间/记录类：`0.1s - 3s`
- 流程/标题类：`0.05s - 3s`
- 推荐类：`0.05s - 3s`
- 总结/开放问答类：允许更慢

## 开发优先级

### 第一阶段：通用检索底座

- 中文 query 拆分
- 标题/文件名召回
- 混合检索
- 知识启用门槛重构
- 低置信度拒答

### 第二阶段：高频快答

- `workflow_steps`
- `person_profiles`
- `worklog_facts`
- 推荐类快答

### 第三阶段：配置化扩展

- 快答配置页面
- 字段别名配置
- 岗位映射配置
- 标题权重配置

## 一句话总结

最终产品形态应是：

`上传解析 -> 文本预览与校对 -> RAG + 结构化双写入库 -> 问题先路由 -> 混合检索 -> 快答优先 -> 模型只做总结与兜底`
